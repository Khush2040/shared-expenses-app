# AI Usage Log

## AI Tools Used
- Google DeepMind Antigravity Coding Agent (Gemini 3.1 Pro High)

## Key Prompts & Architecture
The AI agent was given the initial prompt to build the Shared Expenses App following modern web development guidelines. The prompt structure involved analyzing the CSV to define 15 data anomalies, proposing a database schema, and handling complex interactions without relying on magic numbers.

## Three Concrete Cases Where AI Produced Something Wrong

1. **Incorrect Database Query Syntax for SQLite (mode: insensitive)**
   - **What the AI got wrong**: The AI generated a Prisma query using `mode: 'insensitive'` to do case-insensitive string matching for users: `prisma.user.findFirst({ where: { name: { equals: name, mode: 'insensitive' } } })`.
   - **How it was caught**: The TypeScript compiler failed the `npm run build` process because the `mode` property does not exist in the SQLite adapter for Prisma.
   - **What we changed**: The AI rewrote the logic to fetch the full array of users in memory (`await prisma.user.findMany()`) and performed a case-insensitive `.find()` locally in Node.js instead of trying to push the operation to the SQLite query engine.

2. **Incorrect Prisma Setup for Version 7 vs Version 6**
   - **What the AI got wrong**: The AI initialized the project using `npx prisma init` which defaulted to Prisma v7. It attempted to use `url = "file:./dev.db"` directly in the `schema.prisma` file, which is no longer supported in v7 without specific client constructor logic or configuration files.
   - **How it was caught**: Running `npx prisma db push` threw a schema validation error (`P1012`) stating `url` is no longer supported in schema files.
   - **What we changed**: The AI explicitly downgraded the project dependencies to Prisma v6 (`^6.0.0`) using npm, which allowed the standard `schema.prisma` configuration to work cleanly out of the box without requiring the newly introduced adapter complexities.

3. **Missing Type Annotations inside Map Closures**
   - **What the AI got wrong**: During the anomaly duplicate detection logic, the AI wrote `let existing = anomalies.find(a => a.rowIndex === idx + 1);` without explicitly typing the empty arrays initialized earlier.
   - **How it was caught**: `npm run build` failed the type check due to `existing` implicitly having an `any` type because it was being added to an implicitly typed array `[]`.
   - **What we changed**: Added explicit `any[]` typing to the anomaly arrays `const anomalies: any[] = [];` to satisfy the strict TypeScript compiler configuration generated by Next.js.
